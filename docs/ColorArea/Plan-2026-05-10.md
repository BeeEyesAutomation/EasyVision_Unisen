# Plan & History — ColorArea UI/Engine Refactor (2026-05-10)

## Context
Initial state (CA-001 → CA-006 already shipped):
- Multi mode existed with slot-based assignment (BitwiseOr → connected components → pair list[i] with slot[i]).
- UI had duplicated Tol/Temp `NumericUpDown` inside `layMultiColor` AND main `AdjustBarEx` controls.
- Toggle HSV/RGB cleared color samples.
- After load, Multi mode often showed wrong detection on first inspect (state pollution from legacy `Propety.SetColor()`).
- Set Sample (calibration) only worked correctly for Single.
- Bottom-corner total displayed total detected pixels, not deviation.
- Combined-mask connected-components could merge close colors, mis-pairing lists with regions.
- No order/position validation: a wire swap with unchanged pixel count passed as OK.

User request scope (this iteration):
1. Unify UI controls between Single/Multi (single source of truth: AdjustBarEx).
2. HSV/RGB toggle preserves color list; auto-refresh params on mode switch.
3. Set Sample handles per-list templates in Multi.
4. Bottom corner = total deviation.
5. Fix list 1 always wrong on first inspect after load.
6. Fix saved `PixelTemplate` overwritten by legacy `PxTemp` on load.
7. Per-region NG/OK with mat overlay tinted by status.
8. Switch to per-list scan to avoid cross-color sticking.
9. Order validation: wrong position vs calibrated → NG.
10. Empty list fallback to calibrated centroid (not index-based middle).

## Task History

### CA-007 — Unified Tol/Temp/ColorType controls
**Why:** Two UIs (NumericUpDown + AdjustBarEx) for same field were error-prone and visually inconsistent.

**Changes (`ToolColorArea.cs`):**
- Removed fields `numColorListExtraction`, `numColorListTemplate`.
- Removed `numericRow` block in `InitializeColorAreaMultiUi` (RowCount 8 → 6).
- `layMultiColor` now contains only: Scan combo + ListBox + Add/Copy/Remove + Name + Enabled.
- Tol/Temp solely controlled by `trackPixel`/`AdjValueTemp` (AdjustBarEx) — synced from `LoadSelectedColorListToUi` for Multi, from legacy fields for Single.
- Removed obsolete `ColorListExtractionChanged`/`ColorListTemplateChanged` handlers.

### CA-008 — Toggle HSV/RGB preserves data; refresh on mode switch
**Why:** Users lost samples when toggling color type; mode switch left stale UI state.

**Changes (`ToolColorArea.cs`):**
- `SetSelectedListColorType`: removed `ClearSelectedColorList()` call.
- `btnHSV_Click`/`btnRGB_Click` (Single path): removed `btnDeleteAll.PerformClick()`; only set `Propety.TypeColor` and refresh palette via new helper.
- Added `RefreshSinglePalette()` — rebuilds `listCLShow` from `Propety.HSVs`/`RGBs` based on current `TypeColor`. Hides colors of other type without erasing them.
- Added `RefreshParamsForCurrentMode()` — Single: pulls Extraction/PxTemp/TypeColor from legacy + refresh palette. Multi: delegates to `LoadSelectedColorListToUi`.
- `btnScanSingle_Click`/`btnScanMulti_Click` call this after switching `Propety.CheckMode` and toggling `layMultiColor.Visible`.

### CA-009 — Native instance reset on mode switch & Multi inspect
**Why:** `BeeCpp.ColorArea` (native) holds state across calls. Cross-mode operations polluted the buffer; first inspect after switching often returned wrong masks.

**Changes (`ColorArea.cs` engine):**
- Added public `PrimeNativeColors()`:
  - Single: calls `SetColor()` (legacy buffer).
  - Multi: iterates `ColorLists`, calls private `SetColor(list)` for each enabled list with colors.
- `SetModel()` calls `PrimeNativeColors()` instead of unconditional `SetColor()`.
- `CheckColorMulti` recreates `ColorAreaPP = new BeeCpp.ColorArea()` at the very start of every Multi inspect. Guarantees a clean buffer regardless of prior cross-mode state, prior `PrimeNativeColors` left-overs, or per-list early-return paths inside `SetColor(list)`.

**Changes (`ToolColorArea.cs`):**
- `btnScanSingle_Click`/`btnScanMulti_Click` recreate `Propety.ColorAreaPP = new BeeCpp.ColorArea()` and call `Propety.PrimeNativeColors()`.

### CA-010 — Set Sample per mode + corner shows deviation
**Why:** Multi calibration didn't update UI; corner showed total detected pixels (not lệch).

**Changes (`ToolColorArea.cs`):**
- `ToolColorArea_StatusToolChanged`: when `IsCalib` done, branches by mode.
  - Single: `Propety.PxTemp = Propety.pxRS; AdjValueTemp.Value = ...;` (existing).
  - Multi: `RefreshColorListDisplay()` + `LoadSelectedColorListToUi()` (engine `CompleteMulti` already wrote per-list `PixelTemplate`).

**Changes (`ColorArea.cs` `DrawResult`):**
- Computes `totalDeviation`:
  - Multi: `Σ region.PixelDeviation`.
  - Single: `|pxRS - PxTemp|`.
- Corner label shows `totalDeviation + " Px"` instead of `pxRS + " Px"`.

### CA-011 — Fix `LoadPara` overwriting saved list values
**Bug:** Saved list values (PixelTemplate=2944) appeared as legacy values (15571) after load. Programmatic `AdjValueTemp.Value = Propety.PxTemp` and `trackPixel.Value = Propety.Extraction` triggered `_ValueChanged` handlers, which (in Multi mode) wrote the legacy value into the SELECTED list — corrupting saved data before `UpdateColorAreaMultiUi` re-read it.

**Changes (`ToolColorArea.cs`):**
- `trackPixel_ValueChanged` & `AdjValueTemp_ValueChanged`: early-return `if (_isSyncingColorAreaMultiUi)`.
- In `LoadPara`, wrap the two legacy syncs with `_isSyncingColorAreaMultiUi = true/false` to suppress write-back.

### CA-012 — Per-list direct count rule (replaces slot-based)
**Why:** Combined `BitwiseOr` mask + connected components grouped neighboring colors into one slot. List[i] paired with slot[i] gave wrong counts when colors stuck together.

**New rule (per `AddPerListResults`):**
- Each list independently filtered (`ApplyColorAreaPostProcess` per list mask).
- `PixelCount = Cv2.CountNonZero(list.Mask)`.
- `BoundingBox = Cv2.BoundingRect(list.Mask)`, `Center` = centroid via `Cv2.Moments`.
- `totalPixels = Σ pixelCount`.

**Engine state changes:**
- Added `[NonSerialized] private List<ColorAreaListMask> _listMasksCache;` — kept alive after `CheckColorMulti` returns so `DrawMultiRegionMats` can render each list's mask separately.
- Disposed at start of next `CheckColorMulti`.
- `AddScanSlotResults` retained as dead code (left in place for now; safe to remove later).
- `CompleteMulti`: `owner.ScoreResult = sumDeviation / 10f` (was `maxDeviation`).

**Drawing (`DrawMultiRegionMats`):**
- Iterates `MultiResult.Regions`, looks up `ColorAreaListMask` by `ListIndex`, calls `Draws.DrawMatInRectRotate` with that list's mask + region's OK/NG color.
- Falls back to single-tint `matProcess` overlay if cache missing.

**Per-region label tint (`DrawMultiRegionResults`):**
- `regionColor` computed inside loop from `region.Result`. Fixed pre-existing bug where `backBrush` once turned NG never reset for later regions.

### CA-013 — Order check + center-template fallback
**Why:** A swapped-color setup with same total pixel count would still pass. User wanted position validation. Empty lists also stacked labels at (0,0) by default.

**Model additions (`ColorAreaColorList`):**
- `int OrderTemplate;` — sorted-rank position assigned at calibration.
- `float CenterTemplateX, CenterTemplateY; bool HasCenterTemplate;` — actual centroid stored at calibration (only when `PixelCount > 0`).

**`ColorAreaRegionResult`:**
- Added `int Label` (used briefly during slot-based attempt; retained for backward compatibility, set to 0 in per-list path).

**Engine flow:**
- `CompleteMulti` (calibration branch): `colorList.OrderTemplate = region.Order`; if `PixelCount > 0` also stores `CenterTemplateX/Y` + sets `HasCenterTemplate = true`.
- `CompleteMulti` (normal branch): if `colorList.OrderTemplate > 0 && colorList.OrderTemplate != region.Order` → forces `region.Result = Results.NG` regardless of pixel deviation. Tool overall NG via existing `hasNg` aggregation.
- `AddPerListResults` empty-mask fallback: prefers `CenterTemplate` if `HasCenterTemplate`; otherwise distributes by index along `ScanDirection` (`(i+0.5)/n * dim`).
- `SortRegionResults` invoked after `AddPerListResults` so `Order` reflects sorted rank; fallback centers participate naturally in the sort.
